# Phase 1: Crypto Compatibility

**Goal:** Prove that TypeScript code can produce SPKI DER keys, fingerprints, and DER signatures that are byte-identical to Go's `pkg/keys/canton_keys.go` — and that Canton accepts them.

This is the critical risk reducer. If the TypeScript crypto output doesn't match, nothing else matters.

## What Must Match Byte-for-Byte

There are three crypto operations that must produce identical output in Go and TypeScript:

### 1. SPKI DER Public Key Encoding

**Go reference:** `CantonKeyPair.SPKIPublicKey()` in `pkg/keys/canton_keys.go`

Given a compressed 33-byte secp256k1 public key:
1. Decompress to uncompressed form (65 bytes: `0x04 || x || y`)
2. Wrap in ASN.1 SubjectPublicKeyInfo structure:
   ```asn1
   SEQUENCE {
     SEQUENCE {
       OID 1.2.840.10045.2.1    -- ecPublicKey
       OID 1.3.132.0.10         -- secp256k1
     }
     BIT STRING <uncompressed_point>
   }
   ```
3. DER-encode the structure (~88 bytes)

**TypeScript implementation:** Use `@noble/secp256k1` for point decompression, manual ASN.1 DER encoding (small — the structure is fixed, only the public key bytes vary).

### 2. Fingerprint (Multihash SHA-256)

**Go reference:** `CantonKeyPair.Fingerprint()` in `pkg/keys/canton_keys.go`

Given SPKI DER bytes from step 1:
1. Prepend a 4-byte big-endian purpose value: `0x0000000c` (purpose = 12)
2. SHA-256 hash the concatenation: `sha256(purpose_bytes || spki_der_bytes)`
3. Wrap in multihash format: `0x12 0x20 || hash` (0x12 = SHA-256 algo, 0x20 = 32-byte length)
4. Hex-encode (no `0x` prefix)

**TypeScript implementation:** WebCrypto `crypto.subtle.digest('SHA-256', ...)` for hashing, manual multihash prefix.

### 3. DER-Encoded ECDSA Signature

**Go reference:** `CantonKeyPair.SignHashDER()` in `pkg/keys/canton_keys.go`

Given a 32-byte hash (from Canton's `PreparedTransactionHash`):
1. Sign with ECDSA over secp256k1 — produces (r, s) values
2. Normalize s to low-S form (s <= n/2) per BIP-62 — `crypto.Sign` in go-ethereum does this automatically
3. ASN.1 DER-encode as:
   ```asn1
   SEQUENCE {
     INTEGER r
     INTEGER s
   }
   ```
   Note: DER INTEGERs are signed — prepend `0x00` if high bit is set.

**Important:** The snap receives the 32-byte hash from the server and signs it **directly** — no re-hashing. Canton's `PrepareSubmission` already returns the SHA-256 hash of the prepared transaction.

**TypeScript implementation:** `@noble/secp256k1` `sign(hash, privateKey, { lowS: true })` for signing, manual DER encoding of (r, s).

## Deliverables

### T1. TypeScript Crypto Module

A standalone TypeScript package (will later become the snap's core):

```
canton-snap/
├── src/
│   ├── spki.ts          -- compressedPubKey → SPKI DER bytes
│   ├── fingerprint.ts   -- SPKI DER → multihash fingerprint hex string
│   ├── sign.ts          -- (privateKey, hash) → DER signature bytes
│   └── index.ts         -- re-exports
├── test/
│   ├── vectors.json     -- test vectors generated by Go (see T2)
│   └── crypto.test.ts   -- tests against vectors
├── package.json
└── tsconfig.json
```

**Dependencies:**
- `@noble/secp256k1` — secp256k1 ECDSA signing + point decompression (no native deps, pure JS, audited)
- `@noble/hashes` — SHA-256 (or use WebCrypto `crypto.subtle.digest`)
- No other crypto deps needed

**No MetaMask Snap dependencies yet.** This is pure crypto code that runs in Node.js for testing. Snap scaffolding comes in Phase 2.

### T2. Go Test Vector Generator

A Go program that uses `pkg/keys/canton_keys.go` to produce deterministic test vectors:

```
cmd/generate-test-vectors/main.go
```

For a set of hardcoded test private keys, output a JSON file containing:
- `privateKey` (32 bytes hex)
- `compressedPublicKey` (33 bytes hex)
- `spkiDer` (hex)
- `fingerprint` (hex string, no 0x prefix)
- For each of several test hashes:
  - `hash` (32 bytes hex)
  - `derSignature` (hex)

Example output:
```json
{
  "vectors": [
    {
      "privateKey": "abcd...0001",
      "compressedPublicKey": "02...",
      "spkiDer": "3056...",
      "fingerprint": "1220...",
      "signatures": [
        {
          "hash": "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855",
          "derSignature": "3045..."
        }
      ]
    }
  ]
}
```

**Note on signature determinism:** go-ethereum's `crypto.Sign` uses RFC 6979 (deterministic k-value), so signatures are deterministic for a given (key, hash) pair. `@noble/secp256k1` also uses RFC 6979 by default. The DER signatures should match byte-for-byte.

### T3. Cross-Validation Test

A TypeScript test that loads `vectors.json` and verifies:
1. `spki(compressedPublicKey)` matches expected `spkiDer` — byte-for-byte
2. `fingerprint(spkiDer)` matches expected `fingerprint` — exact string match
3. `signDER(privateKey, hash)` matches expected `derSignature` — byte-for-byte

If any of these fail, the snap will not work with Canton. Fix the TypeScript code until all vectors pass.

### T4. Go Server-Side Verification Function

Add to `pkg/keys/canton_keys.go`:

```go
// VerifyDER verifies a DER-encoded ECDSA signature against a public key and hash.
// Used by the transfer execute endpoint to validate client-provided signatures
// before forwarding to Canton.
func VerifyDER(compressedPubKey []byte, hash []byte, derSig []byte) error
```

This parses the DER signature, recovers r and s, and verifies against the public key. Needed for the server-side signature check flagged in the plan review.

### T5. Canton Integration Test

**Only after T1-T3 pass.** Using a running Canton node (local devnet):

1. Generate a key in TypeScript (or use a test vector key)
2. Register it with Canton via Go (using the SPKI DER and fingerprint from TypeScript)
3. Prepare a transfer via Go (`PrepareTransfer`)
4. Sign the `PreparedTransactionHash` in TypeScript (`signDER`)
5. Execute the transfer via Go (`ExecuteTransfer`) with the TypeScript-produced signature
6. Verify the transfer succeeded

This is the definitive proof that the crypto is compatible end-to-end.

## Known Gotchas

### Low-S Normalization
ECDSA produces two valid signatures for every (key, hash) pair: one with `s` and one with `n - s`. Canton may reject high-S signatures. Both `crypto.Sign` (Go) and `@noble/secp256k1` (TypeScript) normalize to low-S by default, but verify this in cross-validation.

### DER INTEGER Encoding
DER INTEGERs are signed. If the high bit of `r` or `s` is 1, a leading `0x00` byte must be prepended. If leading bytes are `0x00` and the next byte's high bit is 0, the leading zeros must be stripped. Most ASN.1 libraries handle this, but if hand-encoding, test with values where `r[0] >= 0x80`.

### Fingerprint Purpose Prefix
The Go `Fingerprint()` prepends a 4-byte big-endian purpose value (12 = `0x0000000c`) before hashing. This is non-obvious and will cause fingerprint mismatches if missed. The purpose value represents Canton's `HashPurpose.TopologyTransactionSignature`.

### Uncompressed Point Format
SPKI uses the uncompressed EC point format (65 bytes: `0x04 || x || y`), not the compressed 33-byte form. The `@noble/secp256k1` `Point.fromHex(compressed).toRawBytes(false)` returns the uncompressed form.

### Canton PreparedTransactionHash
Canton returns a raw 32-byte hash from `PrepareSubmission`. The snap must sign this directly — do NOT SHA-256 hash it again. The Go code's `SignHashDER` (not `SignDER`) is the correct reference. `SignDER` hashes its input first; `SignHashDER` signs the hash as-is.
